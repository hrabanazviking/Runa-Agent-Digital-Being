## 1. The Verification Paradox

Here is the fundamental problem of this lecture: a self-modifying system produces code that *no human wrote*. The system designed the modification, generated the proof, and verified it — all without human intervention. How can we trust properties of code whose origin is a machine?

This is not merely philosophical. If a system $P$ modifies itself to become $P'$, and the proof that $P'$ satisfies some property $\phi$ was generated by $P$ (or by $P$'s proof-searching subcomponent), then our trust in $\phi(P')$ depends on our trust in the *proof generation process* — which is itself part of $P$, which may have been modified by a previous version of $P$, and so on.

This is the **verification paradox**: to trust self-modified code, we must trust the verification process; but the verification process may itself have been modified.

The resolution, as we'll see, lies in three key ideas: **compositional verification**, **reflection principles**, and **ghost-state invariants**.

## 2. Compositional Verification

### 2.1 The Key Insight

We don't need to verify $P'$ from scratch. We need to verify **only the change** $\delta$ that transforms $P$ into $P'$, and then verify that $\delta$ preserves the relevant properties of $P$.

**Compositional Verification Principle:** If $P \models \phi$ (program $P$ satisfies property $\phi$), and $\delta$ is a modification such that $\{\phi\} \; \delta \; \{\phi \wedge \psi\}$, then $P' = \text{apply}(P, \delta) \models \phi \wedge \psi$.

This reduces the verification problem from "verify the entire modified system" to "verify the modification preserves the invariant." The modification is typically much smaller than the system, so this is a massive reduction.

### 2.2 Difference Verification

A practical instance of compositional verification is **difference verification**: verify only the parts of the system that changed.

Let $P$ and $P'$ be related by a modification $\delta$ that changes a subcomponent $C$ to $C'$. Suppose we can prove:

1. $\text{Behavior}(C') \implies \text{Interface}_C$ — the new component satisfies the interface of the old component.
2. $\text{Behavior}(C')$ includes any new properties $\psi$ we want.

Then, by the compositionality of the system, $P' = P[C \mapsto C']$ satisfies all properties that $P$ satisfies (since $C'$ conforms to $C$'s interface) plus any new properties $\psi$.

This is exactly how the CRISPR-AI Protocol's three-layer verification works: the structural, behavioral, and resource contracts are verified compositionally for each modification.

### 2.3 Interface Contracts

The key abstraction for compositionality is the **interface contract**. An interface contract for component $C$ specifies:

- **Preconditions:** What $C$ assumes about its inputs.
- **Postconditions:** What $C$ guarantees about its outputs.
- **Invariants:** What $C$ maintains about its internal state.

When $C$ is replaced with $C'$, we only need to show that $C'$'s interface contract is **as strong as** $C$'s — i.e., $C'$ makes the same guarantees and requires no additional assumptions.

**Formally:** $C'$ is a **substitute** for $C$ if:

$$\text{Pre}_C \implies \text{Pre}_{C'} \quad \wedge \quad \text{Post}_{C'} \implies \text{Post}_C$$

(C' requires no more than C does, and guarantees at least as much as C does.)

## 3. Reflection Principles

### 3.1 The Problem of Self-Trust

When a self-modifying system proves properties of its own modifications, we face a question of self-trust. Formally: can a formal system $\mathcal{F}$ prove that its own proof verification procedure is correct?

By Gödel's Second Incompleteness Theorem: no, not if $\mathcal{F}$ is consistent.

This seems like a fatal blow. But reflection principles offer a way forward.

### 3.2 Reflection in Type Theory

A **reflection principle** is a schema that allows a formal system to reason about its own provability. The most important for our purposes is the **Löb reflection principle**:

$$\square(\square \phi \to \phi) \to \square \phi$$

In words: if we can prove that "if $\phi$ is provable then $\phi$ is true," then we can prove $\phi$.

Löb's theorem, derived from this principle, is sometimes seen as a paradox (we'll treat it in detail in Lecture 06). But for verification, the constructive content of reflection is more useful than the paradoxical implications.

### 3.3 The Reflection Bridge

The **reflection bridge** (Rao & Mbeki, 2027) is a verified translation between two levels of reasoning:

1. **Object level:** The formal system $\mathcal{F}$ in which the system proves properties about modifications.
2. **Meta level:** The formal system $\mathcal{M}$ in which we (the designers) prove properties about $\mathcal{F}$.

The bridge is a verified transformation:

$$\text{If } \mathcal{M} \vdash \text{Sound}(\mathcal{F}), \text{ then } \forall \phi. \; (\mathcal{F} \vdash \phi \implies \text{True}(\phi))$$

This does *not* require $\mathcal{F}$ to prove its own soundness (which would violate the incompleteness theorem). It requires *us*, working in $\mathcal{M}$, to prove that $\mathcal{F}$ is sound. This is an external argument, and it is precisely the sort of argument that the Rao-Mbeki verified checker provides.

### 3.4 Stratified Trust

The result is a **stratified trust** structure:

- **Level 0 (Hardware):** We trust the physical substrate. (No way around this.)
- **Level 1 (Checker core):** We trust $\mathcal{C}_{\text{RM}}$ because it has a machine-checked correctness proof in $\mathcal{M}$.
- **Level 2 (Proof generation):** We trust the proofs generated by $P$'s proof generator *because* $\mathcal{C}_{\text{RM}}$ verified them.
- **Level 3 (Modified system):** We trust $P'$'s properties *because* they were proven using verified proofs.

Each level's trust depends on the level below, forming a chain. The chain terminates at Level 0 (the hardware), which must be assumed — but this is unavoidable in any verification.

## 4. Ghost-State Invariant Reasoning

### 4.1 The Problem of Invisible State

Self-modifications can change code in ways that are hard to reason about because the *effects* of a change may not be visible in the immediate execution trace. For instance, adding a caching layer to a system changes its asymptotic memory behavior without changing its input-output behavior.

**Ghost state** is a technique from separation logic that addresses this: we augment the program state with *logical* variables that track properties we care about, even if they don't correspond to physically observable state.

### 4.2 Ghost Variables for Safety Invariants

Consider a self-modifying system with invariant $I$: "the system always terminates within $T$ steps." A modification $\delta$ might restructure the code in a way that preserves termination but changes the step bound to $T'$. We can track this with a ghost variable:

$$\text{Ghost}: \; \tau_{\text{step}} \in [0, T_{\max}]$$

At each modification, the proof obligation includes:

$$I(P) \wedge \tau_{\text{step}}(P) \leq T \implies I(P') \wedge \tau_{\text{step}}(P') \leq T$$

The ghost variable $\tau_{\text{step}}$ doesn't exist in the running code — it is a *specification-only* variable that exists solely in the proof. But it allows us to state and prove properties about the modification that would otherwise be inexpressible.

### 4.3 The Frame Rule for Self-Modification

Separation logic's **frame rule** is essential for modular verification of self-modifying systems:

$$\frac{\{P\} \; \delta \; \{Q\}}{\{P * R\} \; \delta \; \{Q * R\}}$$

where $*$ is the separating conjunction. This says: if a modification satisfies $\{P\}\delta\{Q\}$, then it also satisfies $\{P * R\}\delta\{Q * R\}$ for any $R$ that is *unmodified* by $\delta$.

For self-modifying systems, $R$ represents the parts of the system that the modification doesn't touch. The frame rule allows us to verify only the modified part (using the contract from Lecture 02) and automatically carry forward all properties of the unmodified parts.

### 4.4 Example: Modifying a Safety Monitor

Suppose a self-modifying AI system includes a safety monitor $M$ that checks outputs before they are executed. A modification $\delta$ improves the main policy $P$ (but does not touch $M$). Using the frame rule:

$$\frac{\{\text{SafePolicy}(P)\} \; \delta \; \{\text{SafePolicy}(P')\}}{\{\text{SafePolicy}(P) * \text{Monitor}(M)\} \; \delta \; \{\text{SafePolicy}(P') * \text{Monitor}(M)\}}$$

The safety monitor's properties are automatically preserved, without needing to re-verify $M$. This is compositionality in action.

## 5. Verification of Modifications to the Verifier

The deepest challenge: what happens when the modification targets the *proof generator* or the *proof checker* itself? This is where self-modification becomes truly recursive.

### 5.1 The Bootstrap Problem

If the proof checker $\mathcal{C}$ is modified, who checks that the new checker $\mathcal{C}'$ is correct? The obvious answer — "$\mathcal{C}$ checks $\mathcal{C}'$" — creates a circularity.

The resolution: the checker is part of the **trust base** and is *not modified at runtime*. Instead, modifications to the checker go through an *external* verification process (the reflection bridge, working in $\mathcal{M}$). This is the approach CRISPR-AI takes: the Rao-Mbeki checker is immutable during a self-modification run.

### 5.2 Modifications to the Proof Generator

The proof generator, by contrast, *can* be modified. This is a key feature: the system improves its ability to generate proofs over time. The verification of this modification is:

$$\{\text{GeneratesValidProofs}(\Pi)\} \; \delta \; \{\text{GeneratesValidProofs}(\Pi') \wedge \text{Faster}(\Pi')\}$$

Since the *checker* is unchanged, the soundness of the verification pipeline is preserved even as the generator improves. The generator is free to become faster, cleverer, or more powerful — as long as the proofs it produces are still accepted by the unchanged checker.

## 6. The Full Verification Stack

Putting it all together, verification of self-modified code uses a layered approach:

1. **Difference verification:** Only verify what changed.
2. **Interface contracts:** Changed components must satisfy at least the old interface. Unchanged components are carried forward by the frame rule.
3. **Ghost variables:** Track properties that aren't physically observable.
4. **Reflection bridge:** Trust the proof checker by an external argument (not by the system's own self-reference).
5. **Immutable checker:** The proof checker is not modified at runtime; modifications to the checker go through an external process.

This stack gives us:

- **Soundness:** All verified properties actually hold (by the soundness of $\mathcal{C}$ and $\mathcal{M}$).
- **Compositionality:** Properties of each modification compose automatically.
- **Scalability:** Only the changed parts need re-verification.

---

## 7. Key Takeaways

1. The **verification paradox** — trusting code written by machines — is resolved by compositional verification and the reflection bridge.
2. **Difference verification** only checks what changed; compositional rules carry forward unchanged properties.
3. **Interface contracts** make compositionality precise: new components must satisfy at least the old interface.
4. **Ghost variables** from separation logic allow proofs about properties that aren't in the physical state.
5. **The checker stays immutable** at runtime; modifications to the generator are allowed because soundness depends only on the checker.
