## 1. Core idea

A traditional test says "for *this* input, expect *that* output." A **property-based test** says "for *any* input satisfying these constraints, the following *property* should hold." The test framework — **Hypothesis** in Python — *generates* inputs, finds failing cases, and *shrinks* them to minimal counterexamples. Property tests catch bugs that hand-written example tests miss because they explore corners humans don't think of.

For Runa, property-based testing is the right tool for: validating invariants across many randomly-shaped inputs ([[04-preconditions-postconditions-invariants]]), fuzzing Pydantic schemas, testing state machines through random transition sequences, and ensuring that idempotency holds across arbitrary retries.

## 2. Technique / mechanism

**A simple property test:**

```python
from hypothesis import given, strategies as st

@given(items=st.lists(st.integers()))
def test_sort_idempotent(items):
    assert sorted(sorted(items)) == sorted(items)
```

Hypothesis generates many lists of integers, runs the test on each. If any fails, it *shrinks* the input to a minimal failure case.

**Built-in strategies:**

```python
st.integers()                   # any int
st.integers(min_value=0, max_value=100)  # bounded
st.floats(allow_nan=False, allow_infinity=False)
st.text()                       # any string
st.text(alphabet="abc", min_size=1, max_size=10)
st.lists(st.integers(), max_size=100)
st.dictionaries(st.text(), st.integers())
st.tuples(st.integers(), st.text())
st.uuids()
st.datetimes()
st.booleans()
st.none()
st.just(value)                  # always this value
st.sampled_from([a, b, c])      # one of
st.one_of(st.integers(), st.text())  # either type
```

**Composite strategies:**

```python
@st.composite
def episodes(draw):
    return Episode(
        episode_id=draw(st.uuids()),
        text=draw(st.text(min_size=1, max_size=1000)),
        timestamp=draw(st.datetimes()),
    )

@given(ep=episodes())
def test_episode_round_trips(ep):
    serialized = ep.model_dump_json()
    restored = Episode.model_validate_json(serialized)
    assert restored == ep
```

**Pydantic + Hypothesis:**

```python
from hypothesis_pydantic import from_model

@given(ep=from_model(EpisodeWrite))
def test_validation_holds(ep):
    # ep is a valid EpisodeWrite generated by Hypothesis
    serialized = ep.model_dump_json()
    EpisodeWrite.model_validate_json(serialized)  # should succeed
```

**Stateful testing — testing state machines:**

```python
from hypothesis.stateful import RuleBasedStateMachine, rule, invariant, precondition

class TaskLedgerStateMachine(RuleBasedStateMachine):
    def __init__(self):
        super().__init__()
        self.ledger = TaskLedger()
        self.task_ids = []
    
    @rule()
    def queue_task(self):
        task_id = self.ledger.queue(...)
        self.task_ids.append(task_id)
    
    @rule(task_id=st.sampled_from(["one of the queued"]))
    @precondition(lambda self: len(self.task_ids) > 0)
    def start_task(self, task_id):
        self.ledger.start(task_id)
    
    @invariant()
    def no_more_than_max_in_progress(self):
        assert sum(1 for t in self.ledger.tasks if t.status == "in_progress") <= 10

TaskLedgerTest = TaskLedgerStateMachine.TestCase
```

Hypothesis generates sequences of `rule()` calls and checks that all `invariant()`s hold throughout.

**Shrinking:**

```python
@given(st.lists(st.integers()))
def test_sum_positive(items):
    assert sum(items) > 0  # fails for empty list, negative numbers
```

When this fails (e.g., on `[-5, 3, 1]`), Hypothesis shrinks to the *minimal* failing input — likely `[-1]` or `[0]` or `[]`. The minimal case is the bug report.

**Examples for regression:**

```python
@given(st.integers())
@example(0)
@example(-1)
@example(2**32)
def test_with_known_edge_cases(n):
    ...
```

`@example` decorator adds specific cases that must always run, alongside random.

**Settings:**

```python
from hypothesis import settings, HealthCheck

@settings(max_examples=1000, deadline=500)
@given(...)
def test_thorough(...):
    ...
```

`max_examples`: how many inputs to try. `deadline`: per-test ms before timing out.

**Database persistence:**

Hypothesis saves failing examples to `.hypothesis/` directory. Subsequent runs re-test those first.

## 4. Pitfalls and gotchas

- **Slow tests** kill the dev loop. Property tests are slower than unit tests; run sparingly during dev, comprehensively in CI.
- **Non-deterministic randomness without seed.** Hypothesis seeds itself; tests are reproducible by default.
- **External side effects in tests.** Calling network APIs in property tests = bad time. Stub everything.
- **Overly-restrictive strategies.** If your strategy is so narrow that it doesn't explore much, the property test reduces to a single example. Loosen.
- **Overly-permissive strategies.** Generating `text()` includes Unicode horrors. Often what you want; sometimes too much.
- **Shrinking sometimes fails.** Hypothesis tries; not always perfect. Failing examples are still useful unminimised.
- **Stateful tests get slow.** Long rule sequences explode in complexity.

## 5. Applicability to Runa

For **runa.schemas validation**:

- Property test that every Pydantic model round-trips via JSON.
- Property test that validation rejects mutations that violate invariants.

For **state machines** ([[16-state-machines-reliability]]):

- Stateful property test for Skuld's task state machine.
- Stateful property test for circuit breaker states.

For **Muninn invariants**:

- Property test that writing N episodes then querying returns the right ones, regardless of order, content, embedding dimension.

For **idempotency** ([[05-idempotency-design]]):

- Property test that "operation applied N times" == "operation applied once" for idempotent ops.

For **kernel turn invariants**:

- Property test that any sequence of legal Heard inputs produces a valid Replied output (or a recognised error type).

What to avoid:

- Don't property-test the whole kernel end-to-end. Too slow; stub heavily.
- Don't rely solely on Hypothesis. Specific edge-case unit tests still needed.
- Don't ignore Hypothesis examples in `.hypothesis/examples/` — they're recorded failures.

## 6. Open questions

- **The right max_examples per test.** 100 default; for critical code, 1000+. Trade-off with test runtime.
- **Stateful tests at scale.** Hard to write large stateful tests without combinatorial explosion.
- **Cross-system property tests.** Hypothesis is in-process; testing properties across processes/networks is harder.
